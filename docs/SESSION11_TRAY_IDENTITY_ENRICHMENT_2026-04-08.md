## Honest tray identity audit
Current tray identity is still mostly classification + grouping, not deep extraction.

### What currently distinguishes tray types
- `kind` / classifier layer distinguishes:
  - `TrayHousehold`
  - `TrayLot`
  - `TrayRoom`
  - `TrayItem`
- live compatibility also needs to tolerate plain `Household`, `Lot`, `Room`
- `sourceLocation` distinguishes Tray vs Mods
- `bundleName` and `bundleType` provide grouping-style hints when present
- safety notes already identify misplaced tray content

### What grouping signals actually exist
- `bundleName`
- `bundleType`
- `familyHints` (weak, not guaranteed)

These are grouping hints, not strong proof of a true related-file set.

### What related-file relationships actually exist
There is no strong explicit “related tray files” graph in the current data model.
What exists right now is closer to:
- shared grouping label
- family hint
- duplicate pairing signals elsewhere

That means tray UI must not overstate related-file certainty.

### What naming clues exist
- filename stem
- `bundleName`
- subtype / kind labels
- source location

### Known vs inferred
**Known / system-level facts**
- classified type bucket
- where the file is stored
- file path / size / modified / hash
- safety notes already generated by SimSort

**Derived**
- bundle/group labels
- grouping hints

**Inferred**
- simmer-facing tray identity wording
- summary sentences like inactive library object vs misplaced tray content

### Missing data
- deep household member identity
- lot/room names from strong binary extraction
- robust explicit related-file relationship graph
- stronger creator identity for many tray files

## Product language decisions
Tray UI now leans into simmer-facing language instead of weak technical fallback.

### Row language
Tray row clues now prefer:
1. useful grouping hint
2. storage state (`Stored in Tray` / `Stored in Mods`)
3. creator only when there is room and it is useful

This avoids the weaker old `🔖 Tray` / `Misplaced tray` shorthand.

### Tray summary wording
Tray summary copy now uses classification language instead of overstating certainty.
Examples:
- `Classified as Lot, grouped as LooseBlueprint, is sitting in Mods and needs review.`
- `Classified as Household, grouped as OakHousehold, is stored in Tray as library content.`

### Inspector language
Tray inspector now supports tray-specific identity fields and summary copy.
It should answer:
- what kind of tray object is this?
- where is it stored?
- what grouping hint do we have?
- does it need review because it is misplaced?

### More Details language
Tray deep-inspection language now prefers:
- `Tray summary`
- `Grouping hint`
- storage/load context

It avoids implying a real related-file graph where only a grouping hint exists.

## Section architecture changes
### Tray summary / tray context
The tray-specific sheet section now centers on:
- tray type
- storage state
- grouped as (when useful)
- tray summary sentence
- grouping hint (not “related tray files”)
- tray notes when warnings exist
