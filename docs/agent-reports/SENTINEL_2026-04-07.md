- **FIXED: Top gap eliminated** — `6cb226a` and `65c7eca` addressed gap propagation from screen-frame root cause.
- **FIXED: Library table gap at low-row counts** — Multiple commits (`7e45b5e`, `6d7e06a`, `af31433`, `bc7b000`) replaced native table layout with grid-based tbody scroll that grows to fill available space.
- **FIXED: Inspector expand/collapse** — `eb61b5e`, `ee8ad05`, `b3b476c` replaced distorted circular scale buttons with pure glow; collapse UX is now clean.

- **FIXED: Responsive library layout** — `67b0ab8` introduced inspector overlay at ≤1024px, narrower inspector at ≤1280px, `min-width: 0` on workbench-surface, and `overflow: auto` on table-scroll at compact widths.

## Overflow/clipping sanity

- **SANE: Library table** — `.library-table-scroll` gets `overflow: auto` at ≤1024px; type/facts/source columns hide progressively. Horizontal scroll is allowed rather than clipped.
- **SANE: Downloads queue** — `clamp(280px, 38vh, 840px)` queue height uses `38vh` not `dvh`; at typical laptop sizes with browser chrome this may clip ~1–2 queue rows off-screen.
- **POTENTIAL CLIP: Downloads detail panel** — `clamp(360px, min(var(--downloads-detail-width), 32vw))` caps at `32vw`; on a 15" 4K display (3840px wide) this gives ~1229px wide — likely fine, but the `clamp` minimum may be too rigid for ultra-wide 4K in landscape.
- **POTENTIAL CLIP: Settings detail column** — `clamp(280px, 22vw, 420px)` for the right panel at 4K landscape width gives ~845px, which is reasonable. No hard overflow risk.
- **RISK: `calc(100vh - 220px)` hardcoded offsets** — Used for duplicates-queue, audit-stage, organize-preview, downloads-stage-split heights; these assume a fixed OS title-bar height of ~220px combined. On different OS/windowing configs (e.g., frameless window, different DPI scaling) the subtractend can become wrong, causing vertical scroll or clipped bottom edges.

## Regression risks

- **MEDIUM: Multiple `!important` overrides in responsive block** — `8873: width: min(380px, 45vw) !important` and `8876: overflow: auto !important` use `!important` in the ≤1024px library block; if a future component needs local `overflow` control at that breakpoint, it will be overridden.
- **LOW: `overflow-wrap: anywhere` on paths** — Applied to `.folder-path`, `.file-path`, `.preview-row-route`, `.snapshot-meta`, `.queue-main span`, `.detail-row span`; long paths with no spaces will expand container height unpredictably, breaking row-height assumptions in grids.
- **LOW: Monotonic `loadSeqRef` async response discard** — Library's response squashing (`loadSeqRef`) correctly discards stale results; confirmed present but is a subtle pattern — if `setRows` is ever removed from the discard path, stale data could silently display.

## Remaining risks from laptop to 4K

- **MEDIUM: `100dvh` still the only full-height mechanism** — No `@media (height: ...)` adjustment for displays with unusual vertical real estate (e.g., tall 4K monitors at 2160px height); the Home shell caps at `width: min(1880px, 100%)` at ≥3000px/≥1400px but has no corresponding height cap, so hero sections and stacked module bands may stretch awkwardly tall.
- **LOW: Creator-mode sidebar** — `clamp(232px, 15vw, 284px)` for the three-column settings left nav; at 4K width (3840px) this gives 576px — wide for a sidebar nav; combined with the `::before` gradient separator that stretches to `90%` of that width, it may feel disproportionate on ultra-wide displays.
- **LOW: Downloads queue height `38vh`** — At high-DPI or with browser chrome, the queue may clip 1–2 rows; no `min()` floor in the clamp to prevent this on smaller viewports.
- **LOW: No `@media (min-width: 2560px)` breakpoint** — All responsive rules target 1080/1180/1280/1320/1440/1024; nothing specifically handles the 2560px workstation or 3840px 4K range; content may reflow at 1440px and then remain static through to 4K, leaving the middle ground untested.
